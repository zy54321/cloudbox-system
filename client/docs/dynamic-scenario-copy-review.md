# 动态场景客户演示版文案审查（E1）

> 审查依据：`public/config/dynamic_scenarios.json` 中 `engine_failure`、`smoke`、`nav`、`hijack`、`human` 各节点下 `yes`/`no` 分支的 `title`、`phase`、`state`、`summary`、`desc`、`alert`、`events`、`evidence`、`actions`；并兼顾场景级 `name`、各场景 `postReview`（如存在）。  
> **本阶段不修改** `dynamic_scenarios.json`，仅形成优化清单与 E2 风格规则。

## 1. 单发失效（`engine_failure`）

### 1.1 场景级

| 字段路径 | 当前问题 | 建议方向 |
| --- | --- | --- |
| `scenarios.engine_failure.name` | “单发失效”与部分节点用“单发”混用，客户侧可统一为同一技术用语。 | 全场景与标题、告警中统一为“单发失效”或统一简称策略（二选一，并在规则区固化）。 |
| 叙事结构 | 有云匣（yes）在 node 0–3 有节点，无云匣（no）在 node 4 仍有“落地后”步骤；有云匣路径**未配置**对等的落地/收尾节点，客户若对比两条线可能感觉 yes 线“断在协同”而无闭环。 | E2 若允许：为 yes 补对称收尾文案，或在演示脚本中说明“本演示 yes 线止于空地协同阶段”；**仅文档层面**已标为产品叙事缺口。 |

### 1.2 各节点（节选需优化点）

| 字段路径 | 当前问题 | 建议方向 |
| --- | --- | --- |
| `nodes[0].yes.title` | “发动机转速**骤降至**0%”偏戏剧化、绝对化，易引质疑。 | 改为“转速异常下降/不指示”等可演示、克制的工程表述。 |
| `nodes[0].yes.desc` 与 `summary` | 与 `summary` 内容高度重复（均写“发动机→机载感知预警系统”类路径）。 | `summary` 用一句话价值判断（谁发现、链往哪）；`desc` 补一句**业务目的**或差异点，减少复读。 |
| `nodes[0].no.state` | 单独 `MAYDAY` 对非航空客户不友好。 | 保留可保留，建议 `desc` 或 `title` 补一句“国际通用紧急遇险呼叫（示意）”类短释，或 state 用中文+括注。 |
| `nodes[1].yes.title` | 标题为系统名，像目录而非“这一拍发生了什么”。 | 标题写动作或状态，如“机载预警系统已上报特征”等，与 `phase: 识别` 对齐。 |
| `nodes[1].no.title` | 一句过长，信息堆叠。 | 拆为标题短句+`desc` 里写转报过程，便于朗读。 |
| `nodes[2].yes.title` 与 `nodes[2].no.title` | “体系推演/应急处理”与 `nav` 等场景“体系”用语并存，需统一**机构简称**（与 hijack 中“安指/专家组”等对齐）。 | 在统一规则中固定“体系推演系统/专家组”的对外简称表。 |
| `nodes[3].no.summary` | “签派→卫星地面站→机组”链路过简，且“卫站/卫星地面站”与 `nav` 中“卫站”需统一。 | 与 `nav`、`human` 中卫星链表述统一为同一套中文简称。 |
| 多处 `events`/`evidence` | 大量“（示意）”堆叠，略像**调试或占位**说明。 | 保留“演示/示意”时改为每节点**一次**在 `phase` 或文首说清，子字段减少括注重复；或统一为更自然的“演示用摘要”。 |
| 有云匣线整体 | 可加强“**更早发现、上送、专家组与推演联动**”的一句话主线（目前分散在多条 `actions`）。 | 在 1–2 个关键节点 `summary` 中点题，避免口号化，用动词链即可。 |
| 无云匣线整体 | 人工发现、MAYDAY、管制串联清晰；可**轻度**点出时序/协同依赖人工与话音，不夸大。 | 在 `nodes[0].no` 或 `nodes[1].no` 的 `desc` 用中性一句体现“依机组通报与语音链路逐层展开”，避免“滞后”“落后”等贬义词。 |

## 2. 烟雾异味（`smoke`）

### 2.1 场景级

| 字段路径 | 当前问题 | 建议方向 |
| --- | --- | --- |
| `scenarios.smoke.name` | 与 `nodes[0]` _sensor 用语并存时，“异味”在 no 线、“烟雾”在 yes 线，是刻意对比，但客户可能问“两个是不是同一类事件”。 | 在演示口径或某节点 `desc` 用一句说清“传感路径 vs 人工发现路径”，避免被理解为两套无关场景。 |
| `postReview.items` | 当前为空，与 `nav`/`human` 有复盘项不一致。 | E2 可补 2–3 条**中性**事后要点（如记录、调查衔接），与“人为误操作”的复盘**语气区分**开。 |
| 叙事结构 | `nodes[4].yes` 为空，仅有 `no` 落地后步骤；有云匣路径无对称落地文案。 | 同 `engine_failure`，属于对称性与客户预期问题，E2 再决定是否补 yes 线收尾。 |

### 2.2 各节点

| 字段路径 | 当前问题 | 建议方向 |
| --- | --- | --- |
| `nodes[0].yes.summary` | “机载**通感与黑匣**”与产品其他处“通感算”“云匣”等**术语不统一**。 | 与 hijack、人类场景的“通感/云匣/黑匣”在规则表中统一为对外可讲述的一套名称。 |
| `nodes[1].yes.title` | “机载通感**算**系统”与「通感算」连写，口播易结巴；且“**分析确认**”略 AI 式万能 verb。 | 简化为通称+明确动作，如“机载侧根据规则确认二级烟雾告警”等。 |
| `nodes[2].yes.title` | 单句极长，含“汇聚+重构+多方案+仿真”堆叠。 | 标题保留主谓宾；长信息拆入 `desc`，突出**汇聚→推演→回传**节奏（呼应云匣价值）。 |
| `nodes[2].yes.desc` | “并行仿真与结果回传”技术感强。 | 客户语言：谁先看、先出什么、再给谁。 |
| `nodes[3].yes` 与 `nav` 同句 | `title`“地面专家组发送方案至各部门”在 `engine_failure`/`smoke` 多场景**完全重复**，演示易疲劳。 | 同结构保留，但**替换**关键词（烟雾 vs 单发/导航）各写差异半句。 |
| `nodes[4].no` | 末段为数据采集与调查，与“烟雾”**症状**的关联在字面上未回扣。 | `desc` 中补一句**为何**要下载/衔接调查（如保留舱音与告警时序等），用概括语即可。 |
| 无云匣线 | 从乘务内话、检查单、空管、救援到塔台，链路清楚；`nodes[2].no` 到 `nodes[3].no` 时间跨度感靠数字体现，**文案**可点一句“多部门按预案先后衔接”。 | 不贬低：强调**程序、席位、资源出动**，不强调“慢”。 |

## 3. 导航失效（`nav`）

| 字段路径 | 当前问题 | 建议方向 |
| --- | --- | --- |

| --- | --- | --- |
| `nodes[0].yes.desc` | “虚化/闪烁”“融合轨迹在图上生成”**像界面说明/调试**而非客户叙事。 | 改为**现象→机组感知→可上送**一条线，界面词少放，或放 `events`。 |

| `nodes[1].no.desc` | “**手势席**”过于业内行话。 | 改为“管制席位/电子进程单”等客户可理解的等效词，或括注。 |
| `nodes[2].yes.title` 与 `desc` | “体系应急决策推演系统”全名与 `engine_failure` 等**简称不统一**；“地面运营控制中心专家组”首次出现可略冗。 | 与全文件统一**专家组/推演系统**的简称；`desc` 缩句。 |

## 4. 劫机（`hijack`）—— 克制、非戏剧化、非政治化扩写

> 本场景审查重点：术语准确、程序性表述、**避免**煽情词汇、**避免**具体政治与过度细节；非法干扰/劫机类在演示中**依法规与行业惯例的抽象表述**为主。

| 字段路径 | 当前问题 | 建议方向 |
| --- | --- | --- |
| `scenarios.hijack.name` | 单字“劫机”在客户场合可能**过于刺激**；与后续“非法干扰”混用。 | 对客可选用“**非法干扰（劫机类演示）**”或全场景说明为**桌面推演/示意**，E2 二选一。 |
| `nodes[0].yes.title` | “汇聚至通感算”过技术。 | 强调“**机载多源信息汇聚供安全监测**”，系统名可后置。 |
| `nodes[0].yes.desc` | “**飞控**”“合流”等堆叠。 | 拆短句，减少敏感组合词密度；**云匣/数据**侧用中性“记录与上送”类表述。 |
| `nodes[0].no.title` | “**应答机特情码与空地识别**”略像培训标题。 | 改为“应答机特情码设置与空管侧识别”等**动作**型标题。 |
| `nodes[0].no.desc` | 点出 7500 为**示意**合适；避免“**劫机**”重复高亮，可并提“非法干扰类特情**程序**”。 | 保持**程序性**、避免故事化。 |
| `nodes[1].yes.title` | “上送与飞控**回执**”中“回执”易被理解为行政用语。 | 改为“上送与机载**闭环**反馈”或“**确认反馈**”等航空常用表达。 |
| `nodes[1].yes.state` | “通感算**工作中**”像设备状态机显示。 | 改为“**监测/上送**”等对外状态词。 |
| `nodes[1].no.title` / `desc` | “**公安与军指所**”在演示中**敏感**；`summary` 过简。 | 用行业通用表述如“**有关部门按预案接报**”等，**不展开**具体指挥层级细节；`desc` 已较克制，可再减少指挥主体罗列。 |
| `nodes[2].yes.title` 与 `summary` | “指挥链进入推演与**管域**”抽象度高；“**安指—体系推演**”连接符不统一。 | 标题用**可朗读**的短句；`summary` 与 `desc` 避免同一词（管域/推演）无解释重复。 |
| `nodes[2].yes.desc` | “**严重**非法干扰/劫机**特情**”有叠加强调感。 | 留一处定性即可，避免“严重+劫机”双重刺激。 |
| `nodes[2].yes.alert` | “管域**启封**”偏拟物化/游戏化。 | 改为“管域/预案**激活**”等中性动词。 |
| `nodes[2].no.title` 与 `desc` | “**局方**会商”在演示中需**克制**，避免被理解为特定宣传。 | 可改为“**上级主管单位**按预案组织会商”等**程序性**表述，具体实体名由客户现场决定。 |
| `nodes[3].yes.title` 与 `desc` | “**拦截**”在 yes 与 no 多节点重复出现，**戏剧张力**强。 | 统一用**程序类**动名词（**处置/管域/流量**），减少重复渲染；`desc` 中“**拦截/避让**”与敏感操作放一处概括即可。 |
| `nodes[3].no.title` 与 `desc` | 同上；`desc` 中“**本叙事节点为无云匣路径时间轴上末段**”**完全像元数据/调试说明**。 | 必须改为**客户可读**的结语句，或删除，仅留业务描述。 |
| `nodes[4].yes.title` 与 `desc` | “**拦截**”与“**应急响应阶段**”略像剧本旁白。 | 弱化**阶段**感，加强**多部门按预案分工**的陈述。 |
| 有云匣线 | 数据接入、上送、指挥链、方案回传，可对照强调**早汇聚、可推演、方案回传**（不夸大**实战效果**）。 | 用**可验证**的链路动词，避免“决胜/一键”等。 |
| 无云匣线 | 专线通报、多部门会商、管域与流量等**时间长轴**有体现；`nodes[1].no` 的 `summary` 可略补**无自动汇聚**的**中性**说明（不贬低）。 | 用“**依话音与专线逐席通报**”等事实描述即可。 |

## 5. 人为误操作（`human`）—— 突出**纠正闭环**，避免**事故化**

| 字段路径 | 当前问题 | 建议方向 |
| --- | --- | --- |
| `scenarios.human.name` | “**人为误操作**”略偏**定性过重**，易让客户理解为已发生事故。 | 可改为“**操作偏差/CRM 与监控纠偏**”等，或副标题说明**演示**为**偏差被及时制止**。 |
| `postReview.items`（`no` 线） | 多步骤从 QAR 到**讲评/通告**像**事故事后**链条，与**舱内已纠正**的叙事**张力不一致**。 | 调整语气：强调**安全闭环、经验沉淀**，避免“事件升级”**暗示**；可将“**分析/通告**”与**预防再发**绑在一起，弱化“处罚感”（除非产品定位需要）。 |
| `nodes[0].yes.title` | “操作异常**触发**”过泛。 | 与机载产品名结合写清**检测到什么级别**的异常（仍可用示意）。 |
| `nodes[0].yes.desc` | “机载**智能**辅助系统”与 `nav` 等**智能**称呼泛滥，有 AI 味。 | 统一为“**辅助/提示**”类对外名，**智能**可减频。 |
| `nodes[0].no.title` | “**操纵**异常”与操作异常可能**混称**，同一节点多英文缩写需克制。 | 全场景统一**操纵/操作**其一（航空常用“操纵”偏杆舵）；`CRM` 首次括注中文。 |
| `nodes[1].yes.title` 与 `desc` | “**机载智能响应**”口号感；“5G/卫链**等**”略**列举技术栈**像方案书。 | 标题写清**上送与提示**；链路用**天地链路**等概述即可。 |
| `nodes[1].no.title` 与 `desc` | 纠正闭环**明确**，好。 | `actions` 中“**最小必要机动**”保留，整体保持**不扩大事态**的基调。 |
| `nodes[2].yes.desc` 与 `actions` | “管域/流量**等**要素”略空泛。 | 演示场景下写清**与谁协同、解决什么**（一句）。 |
| `nodes[2].no` | 纠正完成再通报空管，**符合闭环**；时间 `120` 与 yes `3` 对比强，**文案**需避免让无云匣线被读成“更糟”。 | 用“**在稳定后按程序补报**”等**程序性**表述，不强调“晚”。 |
| `nodes[3].yes` | “**下送**”与“**回注**”并置，**术语不统一**（上送/下送/回传已多）。 | 统一**下行链路**的对外动词表。 |
| `nodes[3].yes.events` | “**地面闭环更新**”像内部研发用语。 | 改为**地面侧状态更新/处置单更新**等。 |
| `nodes[3].yes.actions` | “机载**可承接**下一步……”略像接口文档。 | 客户语言：**机载按要点提示/供机组执行核实**。 |
| `nodes[4].yes.title` | “**下显**”过业内。 | “显示终端/下屏操作指引”等更可讲解。 |
| 有云匣线 | 从检测、上送、地面评估、专家下送到指引确报，**闭环清晰**，可在一处用短句点题**“更早感知、快协同、可回传纠正”**（与全局八字协调）。 | 与 `smoke`/`nav` 价值表述**同构不同词**。 |
| 无云匣线 | CRM、交叉检查、再通报，突出**不依赖上送**仍可信**人**的闭环。 | 保持尊重**机组资源管理**，避免暗示“只能靠人”贬义。 |

## 6. 跨场景共性问题

| 问题类型 | 表现 | 建议方向（E2） |
| --- | --- | --- |
| 术语不统一 | 通感/通感算/机载感知预警、卫站/卫星地面站/天地链、专家组/地面专家组/运营控制中心专家组。 | 建立**对外简称表**（5–8 个核心词）全文替换策略。 |
| 同一功能多种叫法 | “体系推演系统”全名与简称并存；黑匣/云匣/记录。 | 每概念**一主一备**（主称谓用于 title/summary）。 |
| 过长句子 | `smoke` node2 yes `title`；`hijack` 多段 `desc`。 | **标题 ≤20 字**为宜；细节进 `desc`。 |
| “AI 味” | “分析确认”“智能响应”。 | 改为**规则/程序/特征匹配**等可解释表述（演示可保留“示意”）。 |
| 不可验证大词 | “实时重构完成”“多方案”若无法演示则弱化。 | 与实机/动画能力**对齐**后再写；否则用**演示级**括注**一次/节点**，子字段减少。 |
| 调试/元信息漏网 | `hijack` node3 no `desc` 末句；`nav` 界面技巧堆在 `desc`。 | 全文件扫描**“本叙事/节点/路径”**等元语句，**禁止**出屏。 |
| 重复标题 | 多场景**同一** `title`（专家组下发）。 | 保留结构，**替换半句**场景词。 |
| 示意泛滥 | 每条 `evidence` 都带（示意）。 | 节点级或字段级**统一**一处声明。 |

## 7. 统一文案风格规则（供 E2 修改 `dynamic_scenarios.json`）

1. **对象客户**：非航空与航空兼有时，**第一次**出现缩略语（FMC、CRM、QAR 等）在 `summary` 或 `desc` **括注一句人话**，后续用简称。  
2. **title**：动作或状态，**非**纯系统名录；**避免**“骤降至/一键/严重/重大”等除非与法规一致。  
3. **summary**：一条**业务链**（主语→谓语→对象），8–20 字为主；不重复 `title` 全文。  
4. **desc**：**目的+路径**，可一句或两句；**禁止**元数据（“本节点为无云匣路径”等）。  
5. **state / phase**：**对外状态词**（上送中、协同中、按程序通报），**避免**“工作中、启封、工作中转”等界面向词。  
6. **events / evidence / actions**：**一条一事**；`（示意）` 控制为每**节点**≤2 次，或改文首**单次**说明“以下为演示用摘要”。  
7. **有云匣（yes）价值**：不口号化，用动词链写清 **发现→汇聚/上送→推演/专家→协同下发** 中**至少两步**的差异化。  
8. **无云匣（no）对比**：用**事实**写清 **机组/乘务发现、话音、管制/运控/救援衔接**，不贬损、不虚构劣势；**避免**“滞后、落后、全靠人喊”等词。  
9. **劫机/非法干扰**：**程序、席位、管域/流量**等中性词为主；**减少**“劫机/拦截”在标题中重复；敏感主体用**行业通用抽象表述**，具体名由现场定。  
10. **人为误操作**：贯穿 **发现偏差→立即纠正（CRM/交叉检查）→稳定→按程序补报/记录→（如需）事后再学习**；**不**在叙事前期用词暗示**事故已定责**。  
11. **全文**：术语表驱动替换，**禁止**调试句与**过度技术栈列举**（5G+卫链可合并为“天地链路”等）。
